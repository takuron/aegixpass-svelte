# AegixPass V1 算法核心设计（遗留）

**注意：V1 版本为遗留版本，仅用于向后兼容。新用户请使用 [V2 版本](/algorithm/v2)。

---

## 概述

AegixPass V1 是一种确定性派生密码生成算法，其输出只依赖于输入的主密码、区分密钥和预设配置。该算法不使用任何外部随机源，因此只要输入相同，输出密码总是相同的。

其主要特点和设计原则如下：

- **确定性 (Deterministic)**：算法设计目的为可复现的密码派生过程，其算法仅使用可固定复现的随机队列完成密码生成操作，不涉及任何不依赖任何外部或硬件随机源。
- **不可逆性 (Irreversible)**：通过强大的单向加密哈希函数来处理输入。即使攻击者获得了最终生成的密码和所用的预设配置，也绝对无法反向推导出您的主密码。
- **复杂度保证 (Complexity Guarantee)**：算法确保最终生成的密码中，至少包含一个来自每个指定字符集（charsets）的字符。
- **抗碰撞性 (Collision Resistance)**：主密码、区分密钥或预设配置中任何一个微小的变化，都会通过哈希雪崩效应，导致最终生成的密码发生巨大改变。

## 算法核心输入

算法的运作依赖于以下三个核心输入：

### 主密码 (`password_source`)
用户持有的核心秘密，程序永不存储。

### 区分密钥 (`distinguish_key`)
用于为不同服务（如网站域名或应用名称）生成不同密码的变量，作为区分生成结果的盐值。

### 预设 (`preset`)
一个 JSON 对象，定义了密码生成的所有参数。

#### V1 预设结构

```json
{
  "name": "AegixPass - Default",
  "version": 1,
  "hashAlgorithm": "argon2id",
  "rngAlgorithm": "chaCha20",
  "shuffleAlgorithm": "fisherYates",
  "length": 16,
  "platformId": "aegixpass.takuron.com",
  "charsets": [
    "0123456789",
    "abcdefghijklmnopqrstuvwxyz",
    "ABCDEFGHIJKLMNOPQRSTUVWXYZ",
    "!@#$%^&*_+-="
  ]
}
```

- `length`: 密码总长度。
- `version`: 算法的版本，V1 为 `1`。
- `charsets`: 一个字符串数组，定义了密码应包含的字符分组。例如，数字、小写字母、大写字母和符号。
- `hashAlgorithm`: 用于生成主种子的哈希算法，可选值：`sha256`, `blake3`, `sha3_256`, `argon2id`, `scrypt`。
- `rngAlgorithm`: 确定性随机数生成器算法，目前实现为 `chaCha20`。
- `shuffleAlgorithm`: 洗牌算法，固定为 `fisherYates`。
- `platformId`: 平台ID，作为一个额外可以变动的盐值用于算法使用者做区分。

## 详细算法流程

密码的生成过程严格遵循以下几个关键阶段：

### 阶段 A: 输入验证

在开始计算前，程序会进行严格的输入检查，以避免产生不安全或无效的结果。

- **非空验证**：确保主密码和区分密钥均不为空。
- **长度验证**：确保请求的密码长度 (`length`) 必须大于或等于字符集分组的数量 (`charsets.length`)。这是因为后续步骤需要为每个字符集分组至少选择一个字符。
- **字符集验证**：确保每个字符集分组内都至少包含一个字符。

### 阶段 B: 生成主种子 (Master Seed)

这是整个确定性算法的基石。一个 32 字节的主种子是后续所有伪随机操作的唯一来源。

1. **拼接输入数据**：将所有影响密码生成的参数和用户输入拼接成一个唯一的、格式固定的长字符串。
    - 格式: `"AegixPass_V{version}:{platform_id}:{length}:{password_source}:{distinguish_key}:{charsets_json}"`
    - 示例:
      ```
      AegixPass_V1:aegixpass.takuron.com:16:MySecretPassword123!:example.com:["0123456789","abc...","ABC...","!@#..."]
      ```
    - 这种设计确保了预设中的任何一个参数（甚至是 `charsets` 的顺序）发生变化，都会生成一个完全不同的种子。
    - **注意**：原始密码 `password_source` 直接出现在组合字符串中。

2. **哈希计算**：根据 hashAlgorithm 的类型，使用不同的方法处理上一步的输入数据：

    - **快哈希** (`sha256`, `blake3`, `sha3_256`):
      - 直接对上一步拼接好的 UTF-8 编码字符串进行哈希计算，得到 32 字节的主种子。

    - **慢哈希** (`argon2id`, `scrypt`):
      慢哈希是内存困难型函数，需要额外的盐（Salt）和计算参数来增加破解难度。

      a. **生成确定性盐**：为了保证整个流程的确定性，盐值由 platformId 通过 SHA-256 哈希生成：
         ```
         salt = sha256(platformId)
         ```

      b. **执行密钥派生**：

         - **对于 argon2id**：使用 Argon2id 算法，结合以下参数：
           - `m_cost`: 内存成本 = 19456 KB (19 MiB)
           - `t_cost`: 时间成本 = 2 次迭代
           - `p_cost`: 并行度 = 1 个线程
           - `dkLen`: 输出密钥长度 = 32 字节

         - **对于 scrypt**：使用 Scrypt 算法，结合以下参数：
           - `N`: CPU/内存成本参数 = 2^15 (32768)
           - `r`: 块大小 = 8
           - `p`: 并行化参数 = 1
           - `dkLen`: 输出密钥长度 = 32 字节

3. **获取种子**：将哈希结果作为 32 字节的主种子。

### 阶段 C: 保证每个字符集至少出现一次 (字符集保证)

为了满足现代密码的复杂度要求，算法会确定性地从每个 `charsets` 分组中挑选一个字符，放入最终的密码中。

**V1 方式（可能有偏）**：

1. **分割种子**：将 32 字节的主种子按顺序分割成多个 4 字节（32位无符号整数）的块。

2. **选择字符**：遍历 `charsets` 数组，对于第 i 个字符集分组：
    - 取出主种子中的第 i 个 4 字节块。
    - 将这个块解释为一个无符号 32 位整数（小端序）。
    - 使用这个整数对当前字符集的长度进行取模运算 (`%`)，得到一个索引。
    - 将该索引对应的字符添加到初始密码数组中。

**注意**：这种简单取模方式可能引入统计偏差。V2 已修复此问题，使用无偏的拒绝采样法。

### 阶段 D: 填充密码剩余长度

此时，密码数组中已经包含了满足基本复杂度的字符，接下来需要用更多"随机"字符填充至用户指定的 `length`。

1. **创建确定性 RNG**：使用整个 32 字节主种子来初始化一个确定性的随机数生成器（ChaCha20）。
    - **Key**: 32 字节的主种子
    - **Nonce**: 12 字节的全零数组（确保确定性）
    - **输出**: 4KB 的随机字节流

2. **合并字符集**：将 `charsets` 数组中的所有字符合并成一个大的字符池。

3. **填充字符**：循环填充剩余长度的每一个位置：
    - 从 RNG 中获取一个随机数。
    - 使用无偏的范围随机数生成逻辑 (`secure_random_range_u32`) 选择字符。
    - 将从字符池中选出的字符添加到密码数组中。

### 阶段 E: 最终整体洗牌

为了消除阶段 C 中引入的、保证性字符位置的任何可预测性，需要对整个密码数组进行最后一次确定性的洗牌。

1. **使用同一 RNG 流**：继续使用阶段 D 创建的 RNG 实例进行操作，确保洗牌操作本身也是完全确定和可复现的。

2. **Fisher-Yates 洗牌**：从后向前遍历密码数组，对于每个位置 `i`：
   - 使用 RNG 生成一个 `[0, i]` 范围内的随机索引 `j`
   - 交换位置 `i` 和 `j` 的字符
   - 使用无偏的范围随机数生成逻辑

### 阶段 F: 组合并返回

将最终洗牌后的字符数组组合成一个字符串，并返回给用户。

## 技术实现细节

### 无偏范围随机数生成器 (`secureRandomRange_u32`)

这个函数在 V1 的阶段 D 和阶段 E 中使用，用于从 RNG 中生成无偏的随机数。

```typescript
function secureRandomRange_u32(next_u32: () => number, max: number): number {
    const range = max;
    const U32_MAX = 4294967295;
    const zone = U32_MAX - (U32_MAX % range);

    let v;
    do {
        v = next_u32();
    } while (v >= zone);

    return v % range;
}
```

**工作原理**：
1. `zone` 是 `range` 的最大倍数且小于等于 `U32_MAX`
2. 只有当随机数 `v < zone` 时才被接受
3. 这样 `v % range` 的结果是完全无偏的

### ChaCha20 作为确定性 PRNG

ChaCha20 是一种流密码，在这里被用作确定性的伪随机数生成器：

- **Key**: 32 字节的主种子
- **Nonce**: 12 字节的全零数组（确保确定性）
- **输出**: 加密后的字节流，具有良好的统计随机性

### 字节序处理

所有 32 位整数的读取都使用**小端序 (Little-Endian)**：

```typescript
dataView.getUint32(0, true)  // true 表示小端序
```

这是为了与 Rust 实现保持一致，确保跨平台兼容性。

## V1 与 V2 的主要差异

| 特性 | V1 (遗留) | V2 (推荐) |
|------|-----------|-----------|
| 哈希架构 | 单一哈希 | 双层哈希（快哈希 + 慢哈希） |
| 密码处理 | 原始密码直接参与组合 | 先进行快哈希保护 |
| 慢哈希支持 | 可选（与快哈希二选一） | 强制要求 |
| 字符集索引 | 直接使用种子切片取模（可能有偏） | 无偏拒绝采样法 |
| RNG 初始化时机 | 阶段 C 之后 | 阶段 B 之后立即创建，统一使用 |

**重要提醒**

V1 版本存在以下潜在问题：
1. **密码直接暴露**：原始密码直接出现在组合字符串中
2. **可能的统计偏差**：阶段 C 使用简单取模方式
3. **不一致的 RNG 使用**：阶段 C 使用种子切片，阶段 D/E 才使用 RNG

建议新用户使用 [V2 版本](/algorithm/v2)，它提供了更好的安全性和更一致的跨平台行为。
