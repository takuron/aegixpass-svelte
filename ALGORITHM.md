# AegixPass 算法核心设计

AegixPass 是一种确定性派生密码生成算法，其输出只依赖于输入的主密码、区分密钥和预设配置。该算法不使用任何外部随机源，因此只要输入相同，输出密码总是相同的。

其主要特点和设计原则如下：

- **确定性 (Deterministic)**：算法设计目的为可复现的密码派生过程，其算法仅使用可固定复现的随机队列完成密码生成操作，不涉及任何不依赖任何外部或硬件随机源。
- **不可逆性 (Irreversible)**：通过强大的单向加密哈希函数来处理输入。即使攻击者获得了最终生成的密码和所用的预设配置，也绝对无法反向推导出您的主密码。
- **复杂度保证 (Complexity Guarantee)**：算法确保最终生成的密码中，至少包含一个来自每个指定字符集（charsets）的字符。
- **抗碰撞性 (Collision Resistance)**：主密码、区分密钥或预设配置中任何一个微小的变化，都会通过哈希雪崩效应，导致最终生成的密码发生巨大改变。

---

## 版本概览

AegixPass 目前支持两个算法版本：

| 特性 | V1 (遗留) | V2 (推荐) |
|------|-----------|-----------|
| 哈希架构 | 单一哈希 | 双层哈希（快哈希 + 慢哈希） |
| 密码处理 | 原始密码直接参与组合 | 先进行快哈希保护 |
| 慢哈希支持 | 可选（与快哈希二选一） | 强制要求 |
| 字符集索引 | 直接使用种子切片取模（可能有偏） | 无偏拒绝采样法 |
| 预设字段 | `hashAlgorithm`, `shuffleAlgorithm` | `fastHashAlgorithm`, `slowHashAlgorithm` |

**强烈建议使用 V2 版本**，它提供了更好的安全性和更一致的跨平台行为。

## 算法核心输入

算法的运作依赖于以下三个核心输入：

### 主密码 (`password_source`)
用户持有的核心秘密，程序永不存储。

### 区分密钥 (`distinguish_key`)
用于为不同服务（如网站域名或应用名称）生成不同密码的变量，作为区分生成结果的盐值。

### 预设 (`preset`)
一个 JSON 对象，定义了密码生成的所有参数。

#### V2 预设结构（推荐）

```json
{
  "name": "AegixPass - Default",
  "version": 2,
  "fastHashAlgorithm": "sha256",
  "slowHashAlgorithm": "argon2id",
  "rngAlgorithm": "chaCha20",
  "length": 16,
  "platformId": "aegixpass.takuron.com",
  "charsets": [
    "0123456789",
    "abcdefghijklmnopqrstuvwxyz",
    "ABCDEFGHIJKLMNOPQRSTUVWXYZ",
    "!@#$%^&*_+-="
  ]
}
```

- `length`: 密码总长度。
- `version`: 算法的版本，V2 为 `2`。
- `charsets`: 一个字符串数组，定义了密码应包含的字符分组。例如，数字、小写字母、大写字母和符号。
- `fastHashAlgorithm`: 用于预处理的快哈希算法，可选值：`sha256`, `blake3`, `sha3_256`。
- `slowHashAlgorithm`: 用于主种子生成的内存困难型哈希算法，可选值：`argon2id`, `scrypt`。
- `rngAlgorithm`: 确定性随机数生成器算法，目前实现为 `chaCha20`。
- `platformId`: 平台ID，作为一个额外可以变动的盐值用于算法使用者做区分。

#### V1 预设结构（遗留）

```json
{
  "name": "AegixPass - Default",
  "version": 1,
  "hashAlgorithm": "argon2id",
  "rngAlgorithm": "chaCha20",
  "shuffleAlgorithm": "fisherYates",
  "length": 16,
  "platformId": "aegixpass.takuron.com",
  "charsets": [
    "0123456789",
    "abcdefghijklmnopqrstuvwxyz",
    "ABCDEFGHIJKLMNOPQRSTUVWXYZ",
    "!@#$%^&*_+-="
  ]
}
```

## V2 算法详细流程（推荐）

V2 算法采用了**双层哈希架构**，在安全性和一致性方面相比 V1 有显著提升。

### 阶段 A: 输入验证

在开始计算前，程序会进行严格的输入检查，以避免产生不安全或无效的结果。

- **非空验证**：确保主密码和区分密钥均不为空。
- **版本验证**：确保预设版本为 2。
- **长度验证**：确保请求的密码长度 (`length`) 必须大于或等于字符集分组的数量 (`charsets.length`)。这是因为后续步骤需要为每个字符集分组至少选择一个字符。
- **字符集验证**：确保每个字符集分组内都至少包含一个字符。

### 阶段 B: 生成主种子 (Master Seed)

这是整个确定性算法的基石。V2 采用**双层哈希架构**，提供更强的安全性保护。

#### B.1 快哈希预处理

首先对敏感输入进行快哈希处理：

1. **盐值快哈希**：使用快哈希算法计算 `platformId` 的哈希值
   ```
   fast_hashed_salt = fast_hash(platformId, fastHashAlgorithm)
   ```

2. **密码快哈希**：使用相同的快哈希算法计算原始主密码的哈希值
   ```
   fast_hashed_password = fast_hash(password_source, fastHashAlgorithm)
   ```

3. **密码哈希转十六进制**：将快哈希后的密码转换为 64 个字符的小写十六进制字符串
   ```
   fast_hashed_password_hex = hex(fast_hashed_password)
   ```

#### B.2 组合输入数据

将所有影响密码生成的参数和快哈希处理后的输入拼接成一个唯一的、格式固定的长字符串：

- **格式**: 
  ```
  "AegixPass_V{version}:{platform_id}:{length}:{fast_hashed_password_hex}:{distinguish_key}:{charsets_json}"
  ```

- **示例**:
  ```
  AegixPass_V2:aegixpass.takuron.com:16:5e884898da28047151d0e56f8dc6292773603d0d6aabbdd62a11ef721d1542:example.com:["0123456789","abc...","ABC...","!@#..."]
  ```

**关键点**：
- 原始密码 `password_source` 不再直接出现在组合字符串中，而是被替换为其快哈希值的十六进制表示
- 这种设计提供了额外的安全层，即使慢哈希被破解，攻击者也无法直接获得原始密码

#### B.3 慢哈希生成主种子

使用内存困难型哈希算法（Argon2id 或 Scrypt）生成最终的 32 字节主种子：

- **输入数据**：`input_data`（B.2 中组合的字符串，UTF-8 编码）
- **盐值**：`fast_hashed_salt`（B.1 中快哈希处理后的盐值）

**Argon2id 参数**：
- `m_cost`: 内存成本 = 19456 KB (19 MiB)
- `t_cost`: 时间成本 = 2 次迭代
- `p_cost`: 并行度 = 1 个线程
- `dkLen`: 输出密钥长度 = 32 字节

**Scrypt 参数**：
- `N`: CPU/内存成本参数 = 2^15 (32768)
- `r`: 块大小 = 8
- `p`: 并行化参数 = 1
- `dkLen`: 输出密钥长度 = 32 字节

最终得到 32 字节的 **主种子 (Master Seed)**。

### 阶段 C: 创建确定性 RNG

使用主种子创建一个确定性的随机数生成器 (RNG)，用于后续所有随机操作。

#### C.1 ChaCha20 RNG 初始化

```
nonce = 12 字节的全零数组
random_byte_stream = chacha20(master_seed, nonce, 4096 字节缓冲区)
```

RNG 从 4KB 的随机字节流中持续提取 32 位无符号整数（小端序）。

### 阶段 D: 保证每个字符集至少出现一次 (字符集保证)

为了满足现代密码的复杂度要求，算法会确定性地从每个 `charsets` 分组中挑选一个字符。

**V2 改进**：使用无偏的 `secureRandomRange_u32` 函数生成索引，而不是 V1 中直接使用种子切片的方式。

对于每个字符集分组：
1. 从 RNG 中获取一个随机数
2. 使用**拒绝采样法**（Rejection Sampling）生成一个在 `[0, charset_length)` 范围内的无偏随机索引
3. 将该索引对应的字符添加到密码数组中

**拒绝采样法原理**：
- 计算有效区域阈值：`zone = U32_MAX - (U32_MAX % range)`
- 如果随机数 `v >= zone`，则丢弃该值并重新获取
- 只有当 `v < zone` 时才接受，此时 `v % range` 是完全无偏的

这种方法确保了大字符池中的每个字符被选中的概率完全相等。

### 阶段 E: 填充密码剩余长度

此时，密码数组中已经包含了满足基本复杂度的字符，接下来需要用更多"随机"字符填充至用户指定的 `length`。

1. **合并字符集**：将 `charsets` 数组中的所有字符合并成一个大的字符池。

2. **循环填充**：对于剩余的每一个位置：
   - 从 RNG 中获取一个随机数
   - 使用无偏范围随机数生成逻辑选择一个字符
   - 将字符添加到密码数组中

### 阶段 F: 最终整体洗牌

为了消除阶段 D 中引入的、保证性字符位置的任何可预测性，需要对整个密码数组进行最后一次确定性的洗牌。

1. **使用同一 RNG 流**：继续使用之前创建的 RNG 实例进行操作，确保洗牌操作本身也是完全确定和可复现的。

2. **Fisher-Yates 洗牌**：从后向前遍历密码数组，对于每个位置 `i`：
   - 使用 RNG 生成一个 `[0, i]` 范围内的随机索引 `j`
   - 交换位置 `i` 和 `j` 的字符
   - 使用无偏的范围随机数生成逻辑

### 阶段 G: 组合并返回

将最终洗牌后的字符数组组合成一个字符串，并返回给用户。

## V1 算法详细流程（遗留）

V1 算法使用单一哈希架构，保留此文档仅用于向后兼容。

### 阶段 A: 输入验证

与 V2 类似，但不验证版本号。

### 阶段 B: 生成主种子 (Master Seed)

1. **拼接输入数据**：
   - 格式: `"AegixPass_V{version}:{platform_id}:{length}:{password_source}:{distinguish_key}:{charsets_json}"`
   - **注意**：原始密码直接出现在组合字符串中

2. **哈希计算**：
   - **快哈希** (sha256, blake3, sha3_256): 直接对拼接字符串进行哈希
   - **慢哈希** (argon2id, scrypt): 
     - 盐值 = `sha256(platformId)`
     - 使用慢哈希算法派生主种子

### 阶段 C: 保证每个字符集至少出现一次

**V1 方式（可能有偏）**：
1. 将 32 字节主种子分割成多个 4 字节块
2. 对每个字符集，取出对应的 4 字节块
3. 解释为 32 位无符号整数（小端序）
4. 直接对字符集长度取模得到索引

**注意**：这种简单取模方式可能引入统计偏差，V2 已修复此问题。

### 阶段 D: 创建 RNG 并填充剩余长度

与 V2 类似，但 RNG 在阶段 C 之后才创建。

### 阶段 E: 最终整体洗牌

与 V2 类似。

### 阶段 F: 组合并返回

与 V2 类似。

## V2 安全性优势分析

### 1. 双层哈希保护

V2 的双层哈希架构提供了多重安全保障：

```
原始密码 → 快哈希 → 十六进制字符串 → 组合输入 → 慢哈希 → 主种子
```

- **第一层保护**：即使慢哈希被暴力破解，攻击者得到的只是快哈希值，而非原始密码
- **第二层保护**：快哈希本身也是单向函数，从快哈希值无法反推原始密码
- **第三层保护**：慢哈希的内存困难特性使得大规模暴力破解在经济上不可行

### 2. 无偏随机数生成

V2 在所有随机操作中统一使用 `secureRandomRange_u32` 函数：

- **拒绝采样法**确保每个可能的结果被选中的概率完全相等
- 消除了简单取模运算可能带来的统计偏差
- 保证了跨平台实现的一致性

### 3. 强制慢哈希

V2 要求必须使用内存困难型哈希算法（Argon2id 或 Scrypt）：

- **Argon2id**: 密码哈希竞赛 (PHC) 获胜者，平衡了抗 GPU 和抗侧信道攻击的能力
- **Scrypt**: 设计用于对抗大规模定制硬件攻击，通过高内存需求提高攻击成本

### 4. 统一的 RNG 管理

V2 在生成主种子后立即创建 RNG 实例，并在所有随机操作中统一使用：

- 避免了 V1 中"先用种子切片、后用 RNG"的不一致设计
- 所有随机决策都基于同一个确定性流，确保可复现性
- 代码结构更清晰，减少了实现错误的可能性

## 技术实现细节

### 无偏范围随机数生成器 (`secureRandomRange_u32`)

这是保证跨平台一致性的核心函数，其数学逻辑与 AegixPass Rust 实现完全等价。

```typescript
function secureRandomRange_u32(next_u32: () => number, max: number): number {
    const range = max;
    const U32_MAX = 4294967295;
    const zone = U32_MAX - (U32_MAX % range);

    let v;
    do {
        v = next_u32();
    } while (v >= zone);

    return v % range;
}
```

**工作原理**：
1. `zone` 是 `range` 的最大倍数且小于等于 `U32_MAX`
2. 只有当随机数 `v < zone` 时才被接受
3. 这样 `v % range` 的结果是完全无偏的

**示例**：
- 如果 `max = 3`, `U32_MAX = 10`
- `zone = 10 - (10 % 3) = 9`
- 只有 `v` 在 `[0, 8]` 范围内时才接受
- 此时 `0, 1, 2` 被选中的概率完全均等

### ChaCha20 作为确定性 PRNG

ChaCha20 是一种流密码，在这里被用作确定性的伪随机数生成器：

- **Key**: 32 字节的主种子
- **Nonce**: 12 字节的全零数组（确保确定性）
- **输出**: 加密后的字节流，具有良好的统计随机性

这种设计确保了：
1. 相同的输入总是产生相同的输出
2. 输出流具有密码学安全的随机性
3. 跨语言、跨平台的实现一致性

### 字节序处理

所有 32 位整数的读取都使用**小端序 (Little-Endian)**：

```typescript
dataView.getUint32(0, true)  // true 表示小端序
```

这是为了与 Rust 实现保持一致，确保跨平台兼容性。

## 测试与验证

V2 算法的正确性通过以下测试保证：

1. **确定性测试**：相同输入必须产生相同输出
2. **唯一性测试**：不同输入（即使微小差异）必须产生不同输出
3. **字符集保证测试**：生成的密码必须包含每个字符集中的至少一个字符
4. **边界情况测试**：处理空输入、Unicode 字符、超长输入等边界情况
5. **V1/V2 差异测试**：确保 V1 和 V2 生成不同的密码（即使输入相同）

## 总结

AegixPass V2 算法通过以下特性提供了安全、可靠的密码派生能力：

| 特性 | 实现方式 |
|------|----------|
| **确定性** | ChaCha20 流密码作为 PRNG，固定参数初始化 |
| **不可逆性** | 双层哈希架构（快哈希 + 内存困难型慢哈希） |
| **复杂度保证** | 从每个字符集至少选择一个字符，然后统一洗牌 |
| **无偏随机性** | 拒绝采样法 (`secureRandomRange_u32`) |
| **跨平台一致性** | 严格的字节序处理、与 Rust 实现对齐的算法逻辑 |

V2 是 AegixPass 的推荐版本，提供了最佳的安全性和一致性保障。
